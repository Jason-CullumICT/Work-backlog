3. Executive Summary

  1. No authentication or authorization exists. Every API endpoint is open to any network caller. Work items can be created, modified, deleted, and transitioned by anyone. This is the single highest-impact finding (RED-001, RED-002).
  2. All multi-step database writes lack transactions. Proposals, reviews, transitions, and updates can leave the system in an inconsistent state if a crash or concurrent request occurs. Concurrent requests can bypass the state machine (CHAOS-003–005).
  3. No graceful shutdown or crash handlers. The Node.js process has no SIGTERM handler and no uncaught exception handler. A crash loses in-flight work with no diagnostic logging (CHAOS-001, CHAOS-002).
  4. Database indexes are entirely missing. Every query on every table performs a full table scan. Change history, filtering, and dedup will degrade linearly as data grows (PERF-001, PERF-002).
  5. Frontend/backend types have diverged. No shared types layer exists. Nullable fields, field names, and type shapes differ — causing silent runtime bugs (QO-001, QO-002, QO-005).

9. P1 Findings (8)

P1 NEW RED-001: Complete Absence of Authentication

Source/Backend/src/app.ts

Category: auth-bypass | Confirmation: Confirmed (code path)

Exploit scenario:

  1. Attacker sends GET /api/work-items — receives all work items, no credentials required
  2. Attacker sends DELETE /api/work-items/<uuid> — item permanently deleted
  3. No JWT, session, API key, or any authentication mechanism exists in the entire codebase

Impact: Full read/write/delete access to all data for any network caller

Recommendation: Add JWT Bearer authentication middleware before all /api/* routes

P1 NEW RED-002: No Authorization / Access Control

Source/Backend/src/app.ts, all route files

Category: privilege-escalation | Confirmation: Confirmed (code path)

Exploit scenario:

  1. No concept of users, roles, or permissions exists
  2. Any caller can approve/reject work items, transition pipeline stages, delete items
  3. No ownership model — all items globally mutable

Impact: Any network client has full admin capabilities

Recommendation: Implement RBAC with roles (submitter, reviewer, admin). Protect transition/review/delete endpoints.

P1 NEW RED-003: Prometheus Metrics Publicly Exposed

Source/Backend/src/app.ts:19

Category: domain-specific | Confirmation: Confirmed (code path)

Exploit scenario:

  1. Attacker sends GET /metrics
  2. Receives internal runtime metrics (heap, event loop, routes, status codes)
  3. Reveals infrastructure details useful for targeted attacks

Recommendation: Restrict /metrics to internal network or require auth

P1 NEW QO-001: Frontend WorkItem Type Missing Nullable Fields

Source/Frontend/src/types/index.ts:39,42

Category: architecture-violation

Detail: Frontend declares description: string and queue: string as non-nullable. Backend returns null for these fields on newly-created items. Frontend components will crash or render "null" text.

Recommendation: Create Source/Shared/types.ts as single source of truth. Align nullable fields.

P1 NEW QO-002: Review Type Field Name Mismatch

Source/Frontend/src/types/index.ts:76

Category: architecture-violation

Detail: Backend uses reviewed_at; frontend uses created_at. The frontend field will always be undefined at runtime because the API never sends created_at.

Recommendation: Change frontend Review.created_at to reviewed_at. Audit all usages.

P1 NEW PERF-001: Missing Index on change_history.work_item_id

Source/Backend/src/migrations/001_initial_schema.ts:40-54

Category: missing-index

Detail: FK column work_item_id has no index. Every history fetch is a full table scan. Table grows unboundedly. Also missing on proposals.work_item_id and reviews.work_item_id.

Recommendation: Add indexes: idx_change_history_work_item_id, idx_proposals_work_item_id, idx_reviews_work_item_id

P1 NEW CHAOS-001: No Graceful Shutdown Handlers

Source/Backend/src/server.ts:12

Category: recovery-failure

Detail: No SIGTERM or SIGINT handlers. Process exits hard with no cleanup — in-flight requests dropped, DB connections leaked, SQLite WAL not flushed.

Recommendation: Add signal handlers: server.close() → db.destroy() → process.exit(0)

P1 NEW CHAOS-002: No uncaughtException / unhandledRejection Handlers

Source/Backend/src/server.ts:1

Category: recovery-failure

Detail: Neither handler exists. Bugs outside Express middleware crash silently with no structured log output for post-mortem.

Recommendation: Add process.on('uncaughtException') and process.on('unhandledRejection') with logger.fatal()
