S3 Executive Summary

  1. The workflow feature is incomplete and non-functional end-to-end. Backend CRUD routes for /api/workflows were never implemented — the store and model exist but are not wired to Express. Two of three frontend routes are missing from App.tsx. Users cannot list or create workflows.
  2. Zero authentication or authorization exists on any API endpoint. All 14+ endpoints — including admin actions like approve, reject, dispatch, and delete — are accessible to any unauthenticated network client. Intake webhooks have no signature verification.
  3. All data is lost on backend restart. Both work item and workflow stores are pure in-memory Maps with no persistence. No graceful shutdown handlers exist. No process-level crash handlers are registered.
  4. Frontend error reporting is silently broken. The API client reads body.message but the backend sends body.error, so users never see actual server error messages.
  5. Spec traceability is circular. The specifications contain no formal FR-IDs; the team invented their own numbering that exists only in code comments, making the traceability chain self-referencing rather than spec-anchored.

S9 P1 Findings

SEC-001 — Complete Absence of Authentication on All API Endpoints P1 NEW

Specialist: Red Teamer | Category: auth-bypass | OWASP: A01, A07

File: Source/Backend/src/app.ts:11-31

Description: The Express application mounts all route handlers with zero authentication middleware. No JWT, session, API key, or any auth mechanism exists. CLAUDE.md references login credentials implying auth was expected.

Exploit Scenario: Any HTTP client can read, create, modify, and delete all work items; trigger workflow actions (route, assess, approve, reject, dispatch); read all dashboard data; and inject items via intake webhooks.

Recommendation: Implement authentication middleware (JWT Bearer token) applied globally to /api/* routes. Intake webhooks should use HMAC signature verification.

SEC-002 — Complete Absence of Authorization / Access Control P1 NEW

Specialist: Red Teamer | Category: privilege-escalation | OWASP: A01

File: Source/Backend/src/routes/workflow.ts:30-259, Source/Backend/src/routes/workItems.ts:20-139

Description: No authorization checks exist. Critical state-changing operations (approve, reject, dispatch, delete) have no role checks.

Exploit Scenario: Attacker calls approve + dispatch endpoints to push arbitrary work items through the entire pipeline, bypassing the assessment pod quality gate.

Recommendation: Implement RBAC middleware. Approve/reject/dispatch require manager role. CRUD operations enforce ownership or team membership.

SEC-003 — Unauthenticated Intake Webhooks Without Signature Verification P1 NEW

Specialist: Red Teamer | Category: auth-bypass | OWASP: A07

File: Source/Backend/src/routes/intake.ts:11-55

Description: Zendesk and automated intake endpoints accept POST requests from any source with no authentication or HMAC signature verification.

Exploit Scenario: Attacker floods system with fake work items, causing denial-of-service or wasting team resources.

Recommendation: Implement HMAC signature verification for Zendesk webhook. Require API key for automated endpoint. Add rate limiting.

QUAL-001 / CHAOS-001 — Backend Workflow CRUD Routes Not Implemented P1 NEW

Specialist: Quality Oracle + Chaos Monkey | Category: spec-drift / missing-endpoint

File: Source/Backend/src/app.ts (routes not registered)

Description: The specification defines 6 workflow CRUD endpoints. The store and model layers exist but no Express routes expose them. Frontend hooks call these endpoints, so all workflow pages will receive 404 errors.

Impact: The entire "Create a new Workflow from an image" feature is non-functional end-to-end.

Recommendation: Implement Source/Backend/src/routes/workflowDefinitions.ts with all 6 endpoints, register in app.ts, add route tests.

QUAL-002 / CHAOS-013 — Frontend Routes Missing from App.tsx P1 NEW

Specialist: Quality Oracle + Chaos Monkey | Category: spec-drift / error-handling

File: Source/Frontend/src/App.tsx

Description: Only /workflows/:id is registered. /workflows (list) and /workflows/new (create) are missing, making WorkflowListPage and CreateWorkflowPage unreachable.

Recommendation: Add missing routes. Place /workflows/new before /workflows/:id to avoid matching "new" as an ID.
