Executive Summary

  1. 1. Zero authentication or authorization. Every API endpoint is publicly accessible with no credentials required. Any network-adjacent actor can create, approve, dispatch, and delete all workflow data. This is the single most critical issue — it undermines every other security control.
  2. 2. Webhook intake endpoints have no signature verification. The Zendesk and automated intake routes accept requests from any source with no HMAC validation, enabling data poisoning and queue flooding attacks.
  3. 3. No graceful shutdown or crash recovery. The Node.js process has no SIGTERM/SIGINT handlers and no unhandledRejection guards. Combined with the in-memory-only store, any crash causes total and irreversible data loss of all workflow state.
  4. 4. Unbounded resource consumption throughout. No payload size limits, no pagination caps, no store eviction policy, and full-collection scans on every dashboard request. The system will degrade and eventually OOM under sustained load.
  5. 5. Assessment pod bypass via route override. Any caller can send overrideRoute: "fast-track" to skip the entire assessment pod — the core quality gate that gives this workflow its "self-judging" capability — with no access check.

P1 Findings

P1 RED-001: Complete Absence of Authentication

Category: auth-bypass | Confirmed: code path | Status: NEW

File: Source/Backend/src/app.ts:11-44

Exploit scenario:

  1. Attacker sends POST /api/work-items with no token, session, or API key.
  2. Request passes through only express.json() and debug logging — no auth middleware exists.
  3. Attacker can create, read, modify, approve, reject, dispatch, and delete all workflow data.

Impact: Any anonymous actor on the network has full control of all workflow state.

Recommendation: Add JWT/API-key authentication middleware before all /api/* routes.

P1 RED-002: Complete Absence of Authorization (RBAC)

Category: privilege-escalation | Confirmed: code path | Status: NEW

File: Source/Backend/src/routes/workflow.ts:85-133, 192-259

Exploit scenario:

  1. No roles or permissions enforced on any endpoint.
  2. Any caller can execute admin-level operations: approve, reject, dispatch, delete.

Impact: Full vertical privilege escalation — any actor can execute all workflow operations.

Recommendation: Implement RBAC with roles (submitter, reviewer, dispatcher, admin). Guard privileged endpoints.

P1 RED-003: Unauthenticated Webhook Intake

Category: auth-bypass | Confirmed: code path | Status: NEW

File: Source/Backend/src/routes/intake.ts:11-54

Exploit scenario:

  1. Attacker sends POST to /api/intake/zendesk with arbitrary payload.
  2. No webhook signature, HMAC, or IP allowlist validation.
  3. Attacker floods the queue with fake tickets, causing data poisoning and DoS.

Impact: Data poisoning, denial of service, impersonation of external systems.

Recommendation: Validate Zendesk webhook signature (HMAC-SHA256). Require API key for automated intake. Add rate limiting.

P1 CHAOS-001: No SIGTERM/SIGINT Graceful Shutdown Handlers

Category: recovery-failure | Confirmed: code path | Status: NEW

File: Source/Backend/src/app.ts

Exploit scenario:

  1. SIGTERM sent (container orchestration, deployment, process manager).
  2. Node.js immediately terminates — no connection drain, no cleanup.
  3. In-flight requests are dropped. All in-memory workflow data is lost permanently.

Impact: Total data loss on any process termination. No graceful degradation.

Recommendation: Add process.on('SIGTERM', () => server.close(...)) before app.listen().

P1 CHAOS-002: No unhandledRejection / uncaughtException Guards

Category: error-handling | Confirmed: code path | Status: NEW

File: Source/Backend/src/app.ts

Exploit scenario:

  1. An unhandled promise rejection escapes a route's try/catch.
  2. In Node ≥ 15, the process crashes with no structured log.
  3. No supervisor signal, no audit trail, total data loss.

Impact: Silent crash with data loss and no observability.

Recommendation: Register process.on('unhandledRejection', ...) and process.on('uncaughtException', ...) with structured logging and process.exit(1).
